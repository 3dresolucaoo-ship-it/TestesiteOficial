## RECOMENDACAO DE PROVEDOR

**Resend** (ja configurado no projeto, `services/email.ts` ja usa Resend SDK).

Motivos pra nao mudar:

1. Resend ja esta verificado com SPF/DKIM/DMARC em hayzer.com.br (feito 15/05). Trocar de provedor agora exige re-verificar dominio, recriar registros DNS, testar deliverability de zero.
2. `services/email.ts` ja tem wrapper com graceful degradation (se RESEND_API_KEY ausente, retorna `{ ok: false }` silencioso sem quebrar o signup).
3. Free tier Resend: 3.000 emails/mes. Com 100 leads na waitlist e 3 emails cada = 300 emails totais. Zero custo.
4. API simples, latencia baixa, boa reputacao de deliverability em BR (testado em prod no D+0 welcome 15/05).

**Alternativas que nao recomendo agora:**
- Mailchimp: overkill, drag-and-drop nao combina com o tom maker do CEO, pago a partir de 500 contatos
- Brevo (ex-Sendinblue): bom para sequencias complexas mas adiciona dependencia desnecessaria. Avaliar em Fase 2 se sequencia crescer.
- Amazon SES: barato mas exige configuracao de bounce handling, complaint processing, suppression list. Adiciona ~4h de setup sem ganho real pro volume atual.

## RECOMENDACAO DE GATILHO TECNICO

**Recomendacao: Supabase Edge Function + pg_cron**

### Arquitetura sugerida

```
waitlist_leads.created_at     (timestamp do signup)
         |
         v
pg_cron · job a cada hora · SELECT leads onde D+1/D+3/D+7 ainda nao foi enviado
         |
         v
Supabase Edge Function · hayzer-email-sequence
         |
         v
services/email.ts · sendWaitlistSequence(to, name, day)
         |
         v
Resend API · disparo
         |
         v
UPDATE waitlist_leads SET email_d1_sent_at = now()  (ou d3, d7)
```

### Colunas necessarias em `waitlist_leads`

```sql
ALTER TABLE waitlist_leads ADD COLUMN IF NOT EXISTS email_d1_sent_at timestamptz;
ALTER TABLE waitlist_leads ADD COLUMN IF NOT EXISTS email_d3_sent_at timestamptz;
ALTER TABLE waitlist_leads ADD COLUMN IF NOT EXISTS email_d7_sent_at timestamptz;
```

A query do cron filtra:
- D+1: `created_at <= now() - interval '1 day'` AND `email_d1_sent_at IS NULL`
- D+3: `created_at <= now() - interval '3 days'` AND `email_d3_sent_at IS NULL`
- D+7: `created_at <= now() - interval '7 days'` AND `email_d7_sent_at IS NULL`

### Por que Edge Function + pg_cron, nao Vercel Cron

1. **Sem cold start em sequencia de email.** Edge Functions no Supabase ficam proximas do DB. Vercel Cron acorda o container do Next.js, faz request pro DB, e o container pode ser reiniciado no meio da execucao se demorar mais que o timeout.
2. **A logica de "quem ainda nao recebeu" fica no DB**, que ja tem os dados. Menos roundtrips.
3. **Supabase Edge Functions gratuitas** ate 500.000 invocacoes/mes. Com 100 leads e 3 emails = 300 invocacoes. Zero custo.
4. **Vercel Cron funciona**, mas e mais fragil pra este caso: timeout de 10s em free tier pode cortar o loop se houver muitos leads. Edge Function no Supabase tem timeout de 60s por default e pode ser aumentado.

### Quando Vercel Cron faz mais sentido

Se a logica de template ficar 100% no Next.js (ex: usando React Email pra templates HTML avancados), Vercel Cron e mais simples de manter porque o codigo fica no mesmo repositorio. Para o volume atual (100 leads, templates HTML inline ja prontos em `services/email.ts`), a Edge Function e o caminho mais simples.
