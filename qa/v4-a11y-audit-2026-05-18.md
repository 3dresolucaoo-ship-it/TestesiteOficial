## Issues críticos (P0 — bloqueiam launch)

### A11Y-01 · Hierarquia de heading invertida — `<h2>` antes do `<h1>`
- **Linhas**: 2630 (`<h2 class="greeting-text">`), 2640 (`<h1 class="cover-figure">`).
- **WCAG**: 1.3.1 Info and Relationships (A) + 2.4.6 Headings and Labels (AA).
- **Problema**: O `greeting` ("Boa noite, Gabriel") é renderizado como `<h2>` antes do `<h1>` do cover ("R$ 12.480"). Screen reader anuncia "Heading level 2, Boa noite Gabriel" antes do heading level 1. Quebra o outline do documento.
- **Fix**: Trocar `<h2 class="greeting-text">` por `<p class="greeting-text">` (greeting é apresentação, não heading). OU subir o `<h1>` para abrir a página e demover o cover-figure para `<h2>` apresentando a receita. **Recomendação**: greeting vira `<p>`, mantém `<h1>` no cover-figure.

### A11Y-02 · Faltando skip-to-content link
- **Linha**: 2398 (`<body>` aberto sem skip link).
- **WCAG**: 2.4.1 Bypass Blocks (A).
- **Problema**: Usuário de teclado precisa atravessar TODA sidebar (~13 links nav + brand + user-pill) antes de chegar no conteúdo. No mobile, mobile-toggle + topbar inteiro. Dashboard é página recorrente — usuário enfrenta isso TODA visita.
- **Fix**: Adicionar logo após `<body>`:
  ```html
  <a href="#main-content" class="skip-link">Pular para o conteúdo</a>
  ```
  No CSS, `.skip-link { position: absolute; left: -9999px; }` e `.skip-link:focus { position: fixed; top: 16px; left: 16px; z-index: 50; background: var(--petrol-500); color: #F2EFEA; padding: 10px 16px; border-radius: 8px; }`. No `<main class="main">` adicionar `id="main-content"`.

### A11Y-03 · Ícones-só botões SVG têm `<svg>` sem `aria-hidden`
- **Linhas**: 2493-2497 (mobile-toggle), 2516, 2520, 2525, 2529, 2533, 2541, 2545, 2550, 2558, 2562 (nav-items), 2593 (chev project-switcher), 2602, 2606 (icon-btn topbar).
- **WCAG**: 4.1.2 Name, Role, Value (A) — ruído pro screen reader.
- **Problema**: Os botões/links têm `aria-label` correto, mas os `<svg>` filhos não têm `aria-hidden="true"`. Screen reader pode ler "Pedidos, image, image" ou anunciar shapes desnecessariamente em alguns modos. Felipe deve garantir que todo SVG decorativo dentro de botão labelado seja `aria-hidden="true"`.
- **Fix**: Adicionar `aria-hidden="true"` em TODOS os SVGs dentro de elementos com `aria-label`/`aria-labelledby`. Já está feito em alguns lugares (linhas 2493 e 2609, 2612) — replicar padrão.

### A11Y-04 · `<a href="#">` para itens nav que abrem rotas
- **Linhas**: 2504, 2515, 2519, 2524, 2528, 2532, 2540, 2544, 2549, 2557, 2561, 2788, 2870, 2906.
- **WCAG**: 2.4.4 Link Purpose in Context (A) + 4.1.2 Name, Role, Value (A).
- **Problema**: Todos os links nav e "ver detalhe", "exportar CSV", "ver série completa" são `<a href="#">`. Para a auditoria do mockup é aceitável, mas Felipe ao converter para React **precisa garantir** que cada um vire um `<Link href="/pedidos">`, `<Link href="/producao">`, etc. Sem isso, screen reader anuncia "link, em branco" e Tab+Enter abre `#` que pula pro topo, quebrando a expectativa.
- **Fix**: Pendência explícita pro Felipe — mapear cada href real durante conversão. Documentar TODO no card React: `// TODO: Felipe — substituir href="#" por rota Next.js Link`.

### A11Y-05 · `<div role="button" tabindex="0">` sem listener de teclado
- **Linha**: 2568 (`<div class="user-pill" role="button" tabindex="0">`).
- **WCAG**: 2.1.1 Keyboard (A).
- **Problema**: `role="button"` exige que Enter e Space disparem ação. O HTML mockup não tem handler (esperado — é mockup), mas Felipe deve garantir no React. Sem isso, usuário de teclado consegue dar foco mas não consegue ativar.
- **Fix**: Na conversão React, trocar `<div>` por `<button class="user-pill" type="button">` (button nativo já tem semantics + teclado). Remove `role="button"` e `tabindex="0"` — vira redundante.

### A11Y-06 · Contraste insuficiente: `--fog-400` (#94908A) sobre `--surface-1` no DARK
- **Linhas**: dark mode `--fog-400: #94908A` (linha 46) usado em `.footnote` (1374), `.cover-meta .sep` (844), `.gauge-thresholds` (1714), `.kpi-mini-meta` (1012).
- **WCAG**: 1.4.3 Contrast (Minimum) (AA) — 4.5:1 texto normal, 3:1 texto large.
- **Problema**: #94908A sobre #161B1F (surface-1) = ratio **~5.05:1** — passa AA pra texto normal por POUCO. MAS sobre `#1F262B` (surface-2 hover) = ~4.18:1 — **falha AA**. Pior: a `.footnote` (linha 1374, 13px) é texto pequeno e fica menor que threshold. CEO já tinha subido fog-400 de #5B564D pra #94908A (comentário linha 40-42) — mas precisa subir mais ou parar de usar em texto small.
- **Fix**: Subir `--fog-400` para `#A6A29A` no dark (ratio ~5.8:1 sobre surface-1, ~5.0:1 sobre surface-2). OU restringir uso de fog-400 apenas a texto ≥16px. Recomendação: subir o token. Validar com [WebAIM Contrast Checker](https://webaim.org/resources/contrastchecker/).

### A11Y-07 · Botões `.alert-cta` e `.next-action-cta` — contraste do CTA petrol no light
- **Linhas**: 2197 (`.next-action-cta`), 1920 (`:root[data-theme="light"] .alert-cta { color: #FFFFFF; }`).
- **WCAG**: 1.4.3 Contrast (Minimum) (AA).
- **Problema**: `.next-action-cta` no light mode mantém `color: #F2EFEA` (linha 2195) sobre `background: var(--petrol-500) = #1F7669` — ratio = ~5.4:1 OK. **Mas** `.alert-cta` no light usa `color: #FFFFFF` sobre `var(--ember-500) = #8B5A33` — ratio = ~5.8:1 OK. NO **dark**, `.alert-cta` usa `color: var(--night-950) = #0A0D0F` sobre `--ember-500 = #D08A4A` — ratio = ~10.2:1 OK. Já tá OK na maioria, mas validar `--ember-400: #E0A269` sobre `--night-950` quando hover = ratio 12.4:1 OK. **NA pílula `.streak-pill`**: `color: #1A100A` sobre gradient `#D08A4A→#A86A30` (linha 398-401) — meio do gradient ≈ #BC7A3D, ratio ~7.8:1 OK. Tudo passa, mas Felipe deve validar com axe nas variantes hover.
- **Fix**: Rodar axe-core no componente convertido. Atenção especial ao `.pill-warning` (linha 1935): `color: var(--ember-400)` sobre `rgba(208, 138, 74, 0.16)` no fundo surface-1 — ratio efetivo pode cair abaixo de 4.5:1. Trocar para `var(--ember-300) = #ECC196` no dark.

### A11Y-08 · Charts (donut + bars + gauge) não têm tabela alternativa
- **Linhas**: 2791-2848 (donut), 2872-2903 (bars), 2961-2989 (gauge), 2928-2935 (queue).
- **WCAG**: 1.1.1 Non-text Content (A) + 1.3.1 Info and Relationships (A).
- **Problema**: Os 4 gráficos têm `role="img"` + `aria-label` resumindo (bom), mas usuário cego **não tem como ler valores individuais** se quiser detalhe. WCAG exige equivalente textual. Donut tem legend abaixo com nomes+valores (parcialmente OK), mas bars/gauge/queue não.
- **Fix**: Adicionar `<table class="sr-only">` após cada chart, com dados tabulares. Exemplo:
  ```html
  <table class="sr-only" aria-label="Dados do gráfico de lucro mensal">
    <thead><tr><th>Mês</th><th>Lucro</th></tr></thead>
    <tbody>
      <tr><td>Dezembro</td><td>R$ 4.200</td></tr>
      ... (todos os 6 meses)
    </tbody>
  </table>
  ```
  Bonus: adicionar botão "ver dados em tabela" visível para usuários sighted que preferem leitura — Felipe pode esconder atrás de `<details>`.

## Issues importantes (P1 — fix antes do launch)

### A11Y-09 · Touch target abaixo de 44x44px — botões topbar e theme-toggle
- **Linhas**: 693-694 (`.icon-btn 38x38`), 725-726 (`.theme-toggle 38x38`).
- **WCAG**: 2.5.5 Target Size (AAA, mas é diretriz móvel Hayzer já adota — boa prática).
- **Problema**: Os 3 botões do topbar (Notificações, Buscar, Tema) são 38x38px. Mobile: dedo médio = 9-11mm = ~46px viewport. Toque pode errar. **Mobile-toggle (linha 2294) é 44x44 — OK.** Os 3 do topbar não.
- **Fix**: Subir para 44x44px no mobile via media query:
  ```css
  @media (max-width: 768px) {
    .icon-btn, .theme-toggle { width: 44px; height: 44px; }
  }
  ```

### A11Y-10 · `.user-avatar` (34x34px) é o único elemento clicável da `user-pill`
- **Linhas**: 567-568.
- **WCAG**: 2.5.5 Target Size + 2.1.1 Keyboard.
- **Problema**: O alvo de clique total da user-pill é o `<div>` pai (linha 2568, padding 8px). Tamanho efetivo OK no desktop, mas no mobile + sidebar drawer ainda pode ser apertado.
- **Fix**: Confirmar no React que a área clicável inclui todo o padding. Visualmente OK, só validar.

### A11Y-11 · `.streak-pill` é decorativa mas tem `role="status"` — anúncio repetitivo
- **Linha**: 3185.
- **WCAG**: 4.1.3 Status Messages (AA).
- **Problema**: `role="status"` faz screen reader anunciar conteúdo automaticamente quando muda. Mas o streak não muda em tempo real — é informativo. Anuncia "Streak de uso consecutivo, 12 dias seguidos no controle" no load (pode ser irritante a cada navegação).
- **Fix**: Trocar `role="status"` por nada (deixar como `<div>` normal) ou `role="complementary"`. Manter `aria-label` simplificado: "Streak: 12 dias seguidos".

### A11Y-12 · Cards bento (`<article tabindex="0">`) não têm ação clara ao Enter
- **Linhas**: 2772, 2854, 2911, 2944, 2992, 3049, 3105.
- **WCAG**: 2.1.1 Keyboard (A).
- **Problema**: Cards são focáveis (`tabindex="0"`) mas Enter/Space não dispara nada. Padrão de usuário: foco no card = clique no card = abre detalhe. Hoje o foco só serve pra hover effect (root-hover SVG).
- **Fix**: Felipe define: ou (a) remove `tabindex="0"` se card não é interativo (mas perde hover focus), ou (b) implementa `onKeyDown` que dispara mesma ação do clique no `.card-action` "ver detalhe". Recomendação (b) com link real no React.

### A11Y-13 · `.now-print-progress` aria-label fica no `<div>` interno
- **Linhas**: 3075, 3086, 3097 (`<div class="now-print-progress" aria-label="68% concluído">`).
- **WCAG**: 1.3.1 Info and Relationships (A) + 4.1.2 Name, Role, Value (A).
- **Problema**: `<div>` com `aria-label` mas sem `role` é ignorado por screen readers (genéricos não anunciam div). Precisa `role="progressbar"` + `aria-valuenow` + `aria-valuemin/max`.
- **Fix**:
  ```html
  <div class="now-print-progress" role="progressbar"
       aria-label="Progresso da impressão Vaso Cacto"
       aria-valuenow="68" aria-valuemin="0" aria-valuemax="100">
    <div class="now-print-progress-fill" style="width: 68%;"></div>
  </div>
  ```

### A11Y-14 · `<svg>` charts têm `aria-label` mas SVGs filhos não são `aria-hidden`
- **Linhas**: 2792-2816 (donut SVG), 2872 (bars container), 2962-2978 (gauge SVG).
- **WCAG**: 4.1.2 Name, Role, Value (A).
- **Problema**: O `.donut-svg` tem `role="img"` + `aria-label`. Internamente cada `<circle class="donut-arc">` ALSO tem `role="img"` + `aria-label`. Resultado: screen reader anuncia duas vezes (parent label + child label de cada arc). Sobrecarrega usuário.
- **Fix**: Opção A — manter aria-label só no parent, remover dos arcs (eles ficam aria-hidden, e o label parent já descreve). Opção B — remover aria-label do parent, manter nos arcs (mas usuário tem que navegar dentro do SVG, complicado). Recomendação: **Opção A**, e adicionar `<table class="sr-only">` (ver A11Y-08).

### A11Y-15 · Falta `<header>` semântico no top do `<main>`
- **Linha**: 2584-2625.
- **WCAG**: 1.3.1 Info and Relationships (A).
- **Problema**: O `.topbar` é um `<div>` puro mas funciona como header do conteúdo principal. Não é exigência da AA, mas screen readers oferecem navegação por landmarks — `<header>` ajuda muito.
- **Fix**: Trocar `<div class="topbar">` (linha 2587) por `<header class="topbar">`.

### A11Y-16 · `<aside>` da sidebar tem `aria-label="Navegação principal"` mas é `<aside>`, não `<nav>`
- **Linha**: 2503.
- **WCAG**: 1.3.1 Info and Relationships (A).
- **Problema**: Sidebar é navegação principal — deveria ser `<nav>` (ou ter `<nav>` interno labelado). O `<aside>` semanticamente é "conteúdo tangencial". Tem `<nav>` filhos dentro (linhas 2513, 2538, 2555) — OK, mas o wrapper deveria ser `<nav>` ou pelo menos não `<aside>`.
- **Fix**: Manter `<aside>` se for "complementary" ao main, MAS adicionar `<nav aria-label="Principal">` envolvendo os 3 `<nav>` filhos (que ficariam sub-navs). Mais limpo: trocar wrapper de `<aside>` para `<nav class="sidebar" aria-label="Navegação principal">` e remover os 3 `<nav>` internos (deixar como `<ul>`/`<div>`).

### A11Y-17 · `.next-action-text` não tem `<div>` ou container semântico — `<span class="next-action-label">` + `<p>` soltos
- **Linha**: 2749-2752.
- **WCAG**: 1.3.1.
- **Problema**: Estrutura mistura `<span>` (inline) com `<p>` (block) sem hierarquia clara. Funciona, mas em screen reader o "Próxima ação · sugerida" e o conteúdo podem soar desconectados.
- **Fix**: Envolver em `<div class="next-action-text">` (já existe linha 2749) e considerar trocar o `<span>` label por `<small>` ou usar `<dl><dt><dd>` pattern.

### A11Y-18 · `<svg>` decorativos (branch-divisor, root-canvas, decor-roots, watermark) — verificar
- **Linhas**: 2407 (decor-roots), 2430 (root-canvas), 2489 (watermark), 2714 (branch-divisor).
- **Status**: Todos têm `aria-hidden="true"`. **OK**, só validar que nada quebra no NVDA virtual buffer.

### A11Y-19 · Foco visível pode "comer" 2px dentro do card
- **Linhas**: 201-206 (focus-visible).
- **WCAG**: 2.4.7 Focus Visible (AA).
- **Problema**: `outline-offset: 2px` em `.bento-card:focus-visible` faz outline IR PRA FORA, mas se card é highlighted (com glow box-shadow), pode haver sobreposição visual. Em mobile com cards span-1 que ficam edge-to-edge, outline pode ser cortado.
- **Fix**: Validar visualmente. Se cortar, usar `outline-offset: -2px` (vai pra dentro) ou `box-shadow: 0 0 0 2px var(--petrol-300)` em vez de outline.

### A11Y-20 · Cores indicando estado SEM ícone+texto — `delta.up` / `delta.down` / `delta.neutral`
- **Linhas**: 1158-1180.
- **WCAG**: 1.4.1 Use of Color (A).
- **Problema**: Já tem caracter `▲` (linha 2674, 2685, 2696, 2707) — OK! E o texto "+26,7%" / "−3%" já indica direção. **Mas** o `.kpi-mini-delta.neutral` (linha 2696 "−3% vs sem 02") usa cor neutra `fog-300` — usuário com daltonismo pode confundir com `down` (ember). Texto "neutro" não está claro.
- **Fix**: Adicionar prefixo textual aos deltas: "neutro: −3%", ou trocar caracter "−" por "=" pra deltas neutros (= 3% vs sem 02). Ou simplesmente expandir `aria-label`: `<span class="kpi-mini-delta neutral" aria-label="Variação neutra: redução de 3%">−3% vs sem 02</span>`.

### A11Y-21 · `.queue-bar` com `aria-label="Sábado 4"` é ambíguo
- **Linha**: 2929 (`aria-label="Sábado 4"`).
- **WCAG**: 1.1.1 Non-text Content (A).
- **Problema**: "Sábado 4" pode ser interpretado como "Sábado, dia 4 do mês" em vez de "Sábado, 4 jobs". Screen reader vai ler literalmente.
- **Fix**: Trocar para `aria-label="Sábado: 4 jobs"` (linhas 2929-2935). Mais explícito.

## Como testar pós-implementação

### 1. Ferramentas automatizadas (dev)
- **axe DevTools** (Chrome extension): roda em qualquer página, lista violations. Meta: zero "serious" ou "critical".
- **Lighthouse a11y audit** (Chrome DevTools → Lighthouse → Accessibility): meta ≥ 95.
- **WAVE** (https://wave.webaim.org): segundo opinion, pega coisas que axe não pega.
- **Pa11y CI** (linha de comando): integrar no CI/CD pra rodar a cada PR. `pa11y-ci --sitemap https://hayzer.com.br/sitemap.xml`.

### 2. Screen readers (manual, obrigatório)
- **NVDA** (Windows, free): https://www.nvaccess.org/download/
  - Testar: Tab navega tudo, F leio headings (deve listar h1 do cover → h2 das sections), D lista landmarks (deve ver main, nav, complementary).
- **VoiceOver** (macOS/iOS): Cmd+F5 ou Settings → Acessibilidade → VoiceOver.
  - Testar iOS Safari: rotor → Headings, Links, Landmarks. Navegação por gesto swipe.
- **TalkBack** (Android): Settings → Acessibilidade → TalkBack.
  - Testar Chrome Android: ações por toque + swipe.

### 3. Teclado (manual)
- Tab atravessa toda página sem mouse — todo elemento interativo é alcançável.
- Tab → Enter ativa botões e links.
- Esc fecha drawer mobile (já implementado linha 3378).
- Foco visível em TODOS os elementos focáveis (outline 2px petrol-300).

### 4. Mobile real (obrigatório)
