## GAPS CRITICOS (acao imediata)

### [SEVERIDADE 4] `inventory` — policy `inventory_public_read` sem filtro

```sql
qual = true
```

Qualquer usuario anonimo (ou autenticado de outro tenant) le 100% do inventario de TODOS os makers. Vazamento cross-tenant real. A tabela tem dados sensiveis de estoque, quantidades e custos.

**Fix:**
```sql
DROP POLICY IF EXISTS inventory_public_read ON public.inventory;
-- Se acesso publico for necessario futuramente, recriar com is_public = true
```

---

### [SEVERIDADE 4] `products` — policy `products_public_read` sem filtro

```sql
qual = true
```

Igual ao inventory. Todos os produtos de todos os makers ficam expostos via `/rest/v1/products`. Sem filtro por `is_public` ou `user_id`, qualquer pessoa ve nome, preco, custo de fabricacao e dados de todos os makers.

**Fix:**
```sql
DROP POLICY IF EXISTS products_public_read ON public.products;
CREATE POLICY products_public_read ON public.products
  FOR SELECT TO anon, authenticated
  USING (is_public = true);
```

### [SEVERIDADE 4] Function `process_webhook_atomic` — SECURITY DEFINER callable por anon

Endpoint: `POST /rest/v1/rpc/process_webhook_atomic`

Qualquer pessoa sem autenticacao pode chamar essa funcao e criar **pedidos (orders), transacoes financeiras e ordens de producao** em qualquer `project_id` passado como parametro. Nao exige `auth.uid()` — aceita `p_merchant_id` como UUID externo.

**Fix:**
```sql
REVOKE EXECUTE ON FUNCTION public.process_webhook_atomic(
  text,text,text,jsonb,text,uuid,uuid,text,text,text,
  numeric,text,text,text,text,text,text,text,uuid
) FROM anon;
REVOKE EXECUTE ON FUNCTION public.process_webhook_atomic(
  text,text,text,jsonb,text,uuid,uuid,text,text,text,
  numeric,text,text,text,text,text,text,text,uuid
) FROM authenticated;
```

Esta funcao deve ser chamada exclusivamente via API route server-side (Next.js) com service_role key, nao via Supabase JS client.

### [SEVERIDADE 3] Function `set_active_payment_config` — IDOR via parametro externo

Aceita `p_user_id` como parametro externo. A validacao `v_owner <> p_user_id` verifica se a config pertence ao `p_user_id` passado — mas o caller controla esse valor. Um atacante com UUID de outro user pode explorar via brute-force de config IDs.

**Fix:**
```sql
REVOKE EXECUTE ON FUNCTION public.set_active_payment_config(uuid, uuid) FROM anon;
-- Idealmente: substituir p_user_id por auth.uid() dentro da function
```

---

### [SEVERIDADE 3] Function `rls_auto_enable` — DDL operation callable por anon

Event trigger SECURITY DEFINER exposta via REST. Na pratica pode nao ser invocavel diretamente (event triggers nao aceitam chamada direta), mas o grant de EXECUTE para anon e desnecessario e deve ser revogado.

**Fix:**
```sql
REVOKE EXECUTE ON FUNCTION public.rls_auto_enable() FROM anon;
```

## Observacao multi-tenant: project_id

A maioria das tabelas usa `user_id = auth.uid()` como unico filtro RLS. No modelo atual do Hayzer (1 maker = 1 user = N projetos), isso funciona — o `project_id` e enforcement da aplicacao, nao do banco. Se no futuro houver membros de equipe com acesso a projetos de outros users, o RLS precisara ser extendido com `project_id IN (SELECT ...)`.
